# Развёртывание VPN Panel

## Назначение

Документ определяет правила развёртывания проекта VPN Panel.

Все окружения должны строиться по единым принципам.

---

# Поддерживаемые окружения

Проект предусматривает следующие окружения:

- Development
- Staging
- Production

Каждое окружение имеет собственную конфигурацию.

---

# Архитектура развёртывания

```
Internet

↓

Nginx

↓

Frontend

↓

Backend API

↓

PostgreSQL

↓

HostAgent

↓

VPN Infrastructure
```

Backend является единственной точкой доступа к бизнес-логике.

HostAgent не публикуется наружу.

---

# Контейнеризация

Все компоненты запускаются в Docker.

Каждый сервис содержит собственный Dockerfile.

Все сервисы объединяются через Docker Compose.

---

# Основные сервисы

Минимальный набор сервисов:

- frontend
- backend
- postgres
- nginx
- hostagent

При необходимости могут добавляться:

- prometheus
- grafana
- redis
- pgadmin

---

# Docker Compose

Compose отвечает за:

- запуск сервисов;
- сети;
- тома;
- переменные окружения;
- зависимости между сервисами.

---

# Docker Networks

Рекомендуется использовать отдельные сети:

```
frontend_network

backend_network

internal_network
```

HostAgent подключается только к внутренней сети.

---

# Постоянные данные

В Docker Volume должны храниться:

- база данных;
- конфигурация VPN;
- журналы;
- резервные копии.

Контейнеры должны оставаться максимально статичными.

---

# Переменные окружения

Конфигурация хранится в `.env`.

Запрещается хранить секреты в репозитории.

Минимальный набор:

- DATABASE_URL
- SECRET_KEY
- JWT_SECRET
- HOSTAGENT_URL
- HOSTAGENT_TOKEN
- POSTGRES_USER
- POSTGRES_PASSWORD
- POSTGRES_DB

---

# Reverse Proxy

Nginx отвечает за:

- HTTPS;
- маршрутизацию;
- выдачу статических файлов;
- проксирование запросов к Backend.

---

# HTTPS

В production используется HTTPS.

Сертификаты должны регулярно обновляться.

---

# База данных

Перед запуском Backend необходимо:

1. создать базу;
2. применить миграции Alembic;
3. проверить успешное подключение.

---

# Запуск Backend

Последовательность:

1. загрузка конфигурации;
2. подключение к БД;
3. применение миграций (при необходимости);
4. запуск FastAPI;
5. проверка health endpoint.

---

# Запуск Frontend

Frontend собирается в production-режиме.

Готовая сборка обслуживается через Nginx.

---

# Запуск HostAgent

HostAgent запускается отдельно.

Он должен иметь доступ к:

- VPN-конфигурациям;
- Docker (при необходимости);
- системным сервисам;
- журналам.

Доступ извне запрещён.

---

# Проверка работоспособности

После запуска рекомендуется проверить:

- Frontend;
- Backend;
- PostgreSQL;
- HostAgent;
- доступность API;
- корректность авторизации.

---

# Обновление приложения

Обновление выполняется в следующем порядке:

1. резервное копирование;
2. получение новой версии;
3. пересборка контейнеров;
4. применение миграций;
5. запуск сервисов;
6. проверка работоспособности.

---

# Резервное копирование

Регулярному резервному копированию подлежат:

- PostgreSQL;
- конфигурация VPN;
- файлы конфигурации;
- журналы (при необходимости).

Следует регулярно проверять возможность восстановления.

---

# Мониторинг

Рекомендуется контролировать:

- загрузку CPU;
- использование памяти;
- место на диске;
- состояние контейнеров;
- доступность API;
- время ответа;
- ошибки приложения.

---

# Логирование

Логи рекомендуется разделять по сервисам:

- frontend
- backend
- hostagent
- nginx
- postgres

Следует использовать централизованный сбор логов при росте проекта.

---

# CI/CD

Планируется автоматизация:

- проверки кода;
- запуска тестов;
- сборки Docker-образов;
- публикации артефактов;
- развёртывания.

Развёртывание в production рекомендуется выполнять только после успешного прохождения всех проверок.

---

# Масштабирование

Архитектура должна допускать:

- горизонтальное масштабирование Backend;
- вынос PostgreSQL на отдельный сервер;
- подключение нескольких HostAgent;
- добавление новых VPN-серверов без изменения архитектуры.

---

# Производственная среда

В production необходимо:

- отключить режим debug;
- использовать HTTPS;
- ограничить доступ к административным интерфейсам;
- использовать сильные секреты;
- регулярно обновлять зависимости;
- контролировать состояние сервисов.

---

# Восстановление после сбоя

Рекомендуемый порядок:

1. определить причину;
2. восстановить инфраструктуру;
3. восстановить базу данных;
4. восстановить конфигурацию;
5. проверить доступность сервисов;
6. провести аудит после восстановления.

---

# Заключение

Развёртывание VPN Panel должно быть воспроизводимым, документированным и максимально автоматизированным.

Любое новое окружение должно запускаться по единым правилам, описанным в данном документе.
