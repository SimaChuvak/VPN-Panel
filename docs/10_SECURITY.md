# Безопасность VPN Panel

## Назначение

Документ определяет требования безопасности для всех компонентов проекта VPN Panel.

Все подсистемы проекта должны соответствовать настоящему документу.

---

# Основные принципы

Безопасность строится на принципах:

- минимально необходимых привилегий;
- разделения ответственности;
- защиты по умолчанию (Secure by Default);
- отказоустойчивости;
- аудита действий.

---

# Архитектура безопасности

Все обращения пользователей проходят через Backend.

```
Пользователь

↓

Frontend

↓

Backend

↓

HostAgent

↓

VPN Infrastructure
```

HostAgent не должен быть доступен напрямую из Интернета.

---

# Аутентификация

Для пользователей используется JWT.

После успешного входа выдаются:

- Access Token;
- Refresh Token.

Все защищённые запросы требуют действительного Access Token.

---

# Авторизация

Права определяются ролями и разрешениями.

Доступ к операциям проверяется на стороне Backend.

Frontend не является источником контроля доступа.

---

# Управление ролями

Минимально поддерживаются:

- Administrator;
- Operator;
- Viewer.

Набор ролей может расширяться без изменения архитектуры.

---

# Управление секретами

Запрещается хранить:

- пароли;
- токены;
- API-ключи;
- приватные ключи;
- сертификаты

в исходном коде или репозитории.

Все секреты передаются через:

- переменные окружения;
- защищённые хранилища секретов.

---

# Пароли

Пароли никогда не сохраняются в открытом виде.

Используется современный алгоритм хеширования (например, Argon2id или bcrypt).

---

# Передача данных

Все внешние соединения выполняются по HTTPS.

Незащищённый HTTP допускается только в локальной среде разработки.

---

# Взаимодействие с HostAgent

Backend является единственной системой, имеющей право обращаться к HostAgent.

HostAgent:

- не публикует пользовательский API;
- принимает запросы только от Backend;
- проверяет подлинность запросов.

---

# Проверка входных данных

Все входные данные валидируются через Pydantic.

Запрещается использовать необработанные пользовательские данные без проверки.

---

# Защита API

Следует применять:

- проверку JWT;
- проверку ролей;
- ограничение доступа к административным маршрутам;
- контроль времени жизни токенов.

---

# CORS

Разрешённые источники задаются явно.

Использование значения `*` в производственной среде запрещено.

---

# CSRF

Для cookie-based аутентификации необходимо использовать защиту CSRF.

При использовании Bearer JWT риск существенно снижается, однако следует учитывать сценарии работы браузера.

---

# Ограничение запросов

Для публичных API рекомендуется использовать Rate Limiting.

Это помогает защититься от:

- перебора паролей;
- автоматизированных атак;
- чрезмерной нагрузки.

---

# Логирование

Необходимо фиксировать:

- успешные входы;
- неудачные попытки входа;
- административные действия;
- изменение конфигурации;
- ошибки безопасности.

Запрещается логировать секретные данные.

---

# Аудит

Для критически важных операций должен сохраняться журнал:

- пользователь;
- действие;
- объект;
- время;
- результат;
- IP-адрес (при наличии).

---

# Работа с файлами

Все загружаемые файлы должны проходить проверку:

- размера;
- типа;
- содержимого (при необходимости).

Недопустимо сохранять пользовательские файлы в исполняемых каталогах.

---

# Работа с базой данных

Доступ к базе осуществляется только через слой Repository.

Используются параметризованные запросы ORM.

Следует избегать выполнения необработанного SQL без необходимости.

---

# Защита от распространённых угроз

При разработке необходимо учитывать защиту от:

- SQL Injection;
- XSS;
- CSRF;
- Command Injection;
- Path Traversal;
- SSRF;
- Replay Attack;
- Brute Force.

---

# Обновление зависимостей

Необходимо регулярно обновлять библиотеки и устранять известные уязвимости.

Перед обновлением следует проверять совместимость с проектом.

---

# Резервное копирование

Необходимо предусмотреть:

- резервное копирование базы данных;
- резервное копирование конфигурации;
- проверку возможности восстановления.

---

# Мониторинг

Следует контролировать:

- ошибки авторизации;
- необычную активность;
- отказ сервисов;
- перегрузку системы;
- подозрительные запросы.

---

# Производственная среда

В production запрещается:

- включать режим debug;
- использовать тестовые ключи;
- раскрывать внутренние сообщения об ошибках;
- публиковать служебные эндпоинты.

---

# Реагирование на инциденты

При обнаружении инцидента безопасности рекомендуется:

1. Ограничить воздействие.
2. Зафиксировать события.
3. Проанализировать причину.
4. Устранить проблему.
5. Проверить систему после исправления.
6. При необходимости обновить документацию и процедуры.

---

# Заключение

Безопасность должна учитываться на всех этапах жизненного цикла проекта — от проектирования до эксплуатации.

Каждый новый компонент обязан соответствовать требованиям данного документа.
