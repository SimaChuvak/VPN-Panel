# База данных VPN Panel

## Назначение

Документ определяет правила проектирования базы данных проекта VPN Panel.

Все изменения схемы базы данных должны соответствовать данному документу.

---

# Общие принципы

В проекте используется одна централизованная база данных PostgreSQL.

Доступ к базе осуществляется только через Backend.

Frontend и HostAgent не имеют прямого доступа к БД.

---

# Используемая СУБД

PostgreSQL 17

Используется как единственная поддерживаемая система управления базами данных.

---

# ORM

SQLAlchemy 2.x

Все модели описываются через Declarative Mapping.

Используется исключительно асинхронный режим работы.

---

# Миграции

Alembic

Любое изменение структуры БД должно сопровождаться новой миграцией.

Редактирование уже применённых миграций запрещено.

---

# Основные правила проектирования

## Первичные ключи

Во всех основных таблицах используются UUID.

Пример:

```
id UUID PRIMARY KEY
```

UUID генерируется на стороне приложения.

---

## Именование таблиц

Используется:

snake_case

множественное число.

Примеры:

```
users

vpn_servers

vpn_clients

roles

permissions

audit_logs
```

---

## Именование столбцов

Используется:

snake_case

Примеры:

```
created_at

updated_at

last_login

display_name

server_id
```

---

## Временные поля

Каждая сущность должна иметь:

```
created_at

updated_at
```

При необходимости:

```
deleted_at
```

для реализации мягкого удаления.

---

## Внешние ключи

Имена должны быть очевидными.

Например:

```
user_id

server_id

role_id

client_id
```

---

# Индексы

Индексы создаются:

- для внешних ключей;
- для полей поиска;
- для уникальных значений;
- для часто используемых фильтров.

Избыточные индексы не допускаются.

---

# Уникальные ограничения

Используются только там, где это необходимо.

Например:

```
email

username

server_name
```

---

# Null значения

NULL допускается только при наличии обоснования.

Предпочтение отдаётся NOT NULL.

---

# Soft Delete

При необходимости используется мягкое удаление.

Записи помечаются полем:

```
deleted_at
```

Физическое удаление выполняется только в исключительных случаях.

---

# Связи

Используются:

- One-To-One
- One-To-Many
- Many-To-Many

Связи должны быть явно описаны.

---

# Каскадные операции

Используются только при наличии архитектурного обоснования.

Следует избегать случайного удаления связанных данных.

---

# История изменений

Для критически важных сущностей рекомендуется использовать журнал изменений.

Например:

- пользователи;
- роли;
- VPN-клиенты;
- VPN-серверы.

---

# Аудит

Критические действия должны фиксироваться.

Минимальный набор данных:

- пользователь;
- действие;
- объект;
- время;
- IP (при наличии);
- результат.

---

# Производительность

Следует избегать:

- N+1 запросов;
- лишних JOIN;
- избыточной денормализации.

Использовать eager loading только при необходимости.

---

# Транзакции

Все операции изменения данных выполняются внутри транзакций.

При ошибке транзакция должна быть полностью откатана.

---

# Версионирование схемы

Все изменения схемы фиксируются через Alembic.

Структура базы данных должна быть воспроизводимой с нуля.

---

# Документирование моделей

Каждая новая сущность должна иметь:

- описание назначения;
- связи с другими сущностями;
- ограничения;
- особенности использования.

---

# Планируемые основные сущности

На текущем этапе планируются следующие доменные сущности:

- Users
- Roles
- Permissions
- Sessions
- VPN Servers
- VPN Clients
- VPN Configurations
- Host Agents
- Audit Logs
- System Settings
- Notifications

Конкретная структура каждой сущности будет описана в отдельных архитектурных документах перед реализацией.

---

# Запрещается

Запрещается:

- обращаться к БД напрямую вне слоя Repository;
- изменять структуру таблиц без миграций;
- использовать "магические" SQL-запросы без необходимости;
- хранить бизнес-логику в моделях ORM;
- нарушать соглашения по именованию.

---

# Заключение

Схема базы данных должна быть последовательной, масштабируемой и удобной для сопровождения.

Все новые сущности должны соответствовать правилам, описанным в данном документе.
