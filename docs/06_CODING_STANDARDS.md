# Стандарты кодирования VPN Panel

## Назначение

Документ определяет единый стиль написания кода для Backend, Frontend и HostAgent.

Соблюдение данных стандартов обязательно для всех новых модулей проекта.

---

# Общие принципы

Код должен быть:

- понятным;
- читаемым;
- предсказуемым;
- легко тестируемым;
- легко расширяемым.

Предпочтение отдаётся простым решениям.

---

# Именование

Используется:

snake_case

```
vpn_clients
user_management
```

Используется:

snake_case.py

```
user_service.py

auth_router.py
```

```
create_user()
```

snake_case

# Backend

Backend реализуется слоями.

```
API

↓

Services

↓

Repositories

↓

Database
```

Запрещается нарушать эту последовательность.

---

# API

Контроллеры должны:

- принимать запрос;
- валидировать входные данные;
- вызывать Service;
- возвращать ответ.

В контроллерах запрещается размещать бизнес-логику.

---

# Services

Service отвечает за:

- бизнес-логику;
- проверки;
- вычисления;
- координацию работы нескольких Repository.

Service не должен выполнять SQL-запросы напрямую.

---

# Repository

Repository отвечает только за:

- чтение данных;
- запись данных;
- обновление данных;
- удаление данных.

Repository не содержит бизнес-логики.

---

# SQLAlchemy

Используется современный стиль SQLAlchemy 2.x.

Модели размещаются только в каталоге `models`.

Работа с БД осуществляется через Repository.

---

# Pydantic

Все входные и выходные данные описываются схемами.

Использование словарей вместо схем допускается только в исключительных случаях.

---

# Обработка ошибок

Ошибки должны быть информативными.

Не допускается:

```
except:
    pass
```

Следует перехватывать только ожидаемые исключения.

---

# Логирование

Логи должны содержать:

- уровень;
- сообщение;
- контекст;
- идентификаторы объектов (при необходимости).

Запрещается логировать:

- пароли;
- токены;
- секретные ключи;
- персональные данные без необходимости.

---

# Импорты

Порядок импортов:

1. Стандартная библиотека Python.
2. Сторонние библиотеки.
3. Внутренние модули проекта.

Импорты должны быть сгруппированы.

---

# Frontend

Используется TypeScript.

Новые компоненты должны быть функциональными.

Class Components не используются.

---

# React

Каждый компонент должен иметь одну ответственность.

Следует избегать чрезмерно больших компонентов.

---

# UI

Компоненты интерфейса должны переиспользоваться.

Не допускается копирование одинакового кода в нескольких местах.

---

# Комментарии

Комментарии пишутся только тогда, когда код невозможно сделать самодокументируемым.

Предпочтение отдаётся понятным именам функций и переменных.

---

# Магические значения

Не допускается использование "магических чисел" и строк.

Следует использовать именованные константы.

---

# Асинхронность

Все операции ввода-вывода должны использовать async/await.

Блокирующие вызовы следует избегать.

---

# Тестируемость

Код должен быть написан так, чтобы его можно было тестировать изолированно.

Следует минимизировать жёсткие зависимости между компонентами.

---

# Документирование

Для публичных функций, классов и модулей рекомендуется использовать docstring.

Документация должна описывать назначение, параметры и возвращаемое значение.

---

# Форматирование

Python:

- Black
- Ruff

TypeScript:

- ESLint
- Prettier

Любой код должен проходить автоматическую проверку форматирования.

---

# Запрещается

Запрещается:

- нарушать архитектурные слои;
- использовать глобальное состояние без необходимости;
- дублировать код;
- хранить секреты в исходном коде;
- добавлять временные решения без пометки и последующего устранения.

---

# Заключение

Код проекта должен быть единообразным независимо от автора.

При возникновении неоднозначных ситуаций предпочтение отдаётся читаемости, простоте сопровождения и соблюдению архитектурных принципов.
