# Правила разработки VPN Panel

## Назначение

Документ определяет единые правила разработки проекта.

Все изменения в проекте должны соответствовать данным правилам.

---

# Основные принципы

Приоритеты разработки:

1. Безопасность
2. Простота сопровождения
3. Читаемость кода
4. Масштабируемость
5. Производительность

---

# Жизненный цикл разработки

Каждая новая задача проходит одинаковые этапы.

```
Идея

↓

Анализ

↓

Обсуждение

↓

Архитектурное решение

↓

Обновление документации

↓

Разработка

↓

Тестирование

↓

Code Review

↓

Merge
```

---

# Правило "Документация прежде кода"

Любое архитектурное изменение сначала фиксируется в документации.

Только после этого допускается изменение кода.

Если изменение не требует архитектурного решения (например, исправление локальной ошибки), обновление документации не обязательно.

---

# Работа с Git

Основная ветка:

```
main
```

Разработка новой функциональности выполняется в отдельных ветках.

Пример:

```
feature/auth

feature/dashboard

feature/vpn-clients
```

Исправления ошибок:

```
fix/login

fix/docker

fix/api
```

---

# Pull Request

Каждый Pull Request должен:

- иметь понятное название;
- описывать внесённые изменения;
- ссылаться на соответствующую задачу (при наличии);
- не содержать несвязанных изменений.

---

# Размер изменений

Предпочтение отдаётся небольшим изменениям.

Один Pull Request — одна логическая задача.

Следует избегать крупных изменений, затрагивающих несколько подсистем одновременно.

---

# Code Review

Перед объединением изменений необходимо проверить:

- корректность архитектуры;
- соответствие документации;
- стиль кода;
- отсутствие дублирования;
- отсутствие лишних зависимостей;
- прохождение тестов.

---

# Документация

Документация является частью проекта.

При изменении:

- структуры каталогов;
- архитектуры;
- API;
- базы данных;
- процессов разработки;

необходимо обновить соответствующие документы в каталоге `docs/`.

---

# Работа с зависимостями

Добавление новой библиотеки допускается только если:

- существующий стек не решает задачу;
- библиотека активно поддерживается;
- есть понятная документация;
- лицензия совместима с проектом.

Перед добавлением зависимости следует оценить её влияние на проект.

---

# Рефакторинг

Рефакторинг выполняется только при наличии одной из причин:

- повышение читаемости;
- устранение технического долга;
- подготовка к новой функциональности;
- повышение производительности.

Рефакторинг не должен изменять поведение системы без отдельного решения.

---

# Совместимость

Новая функциональность не должна нарушать работу существующих модулей.

При необходимости изменения публичного API следует предусмотреть стратегию миграции.

---

# Работа с ошибками

При обнаружении ошибки рекомендуется следующий порядок:

1. Воспроизвести проблему.
2. Определить причину.
3. Исправить причину, а не только проявление.
4. Добавить тест (если применимо).
5. Обновить документацию при необходимости.

---

# Безопасность

При разработке необходимо учитывать:

- проверку входных данных;
- обработку исключений;
- принцип минимально необходимых привилегий;
- защиту конфиденциальных данных;
- отсутствие секретов в репозитории.

---

# Логирование

Логи должны быть:

- информативными;
- структурированными;
- без конфиденциальных данных.

Каждое сообщение должно помогать диагностике проблемы.

---

# Тестирование

Перед объединением изменений рекомендуется выполнить:

- статический анализ;
- модульные тесты;
- интеграционные тесты (при наличии);
- ручную проверку новой функциональности.

---

# Запрещается

Без отдельного обсуждения запрещается:

- изменять архитектуру проекта;
- обходить принятые слои приложения;
- напрямую обращаться к базе данных вне слоя Repository;
- добавлять зависимости без обоснования;
- смешивать несколько задач в одном Pull Request;
- хранить секреты, пароли и ключи в репозитории.

---

# Поддержание качества

Каждый новый модуль должен:

- соответствовать архитектуре проекта;
- использовать утверждённый технологический стек;
- сопровождаться понятным кодом;
- быть документированным при необходимости;
- не нарушать существующие соглашения.

Разработка проекта должна оставаться последовательной, предсказуемой и удобной для дальнейшего сопровождения.
